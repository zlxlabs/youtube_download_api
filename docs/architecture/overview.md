## 核心组件

### 1. API Routes 层

**职责**：
- 处理 HTTP 请求和响应
- 请求参数验证
- 鉴权和权限控制
- 调用业务服务层

**主要模块**：
- `routes.py` - 主要 API 路由
- `settings_routes.py` - 设置相关路由
- `manual_upload_routes.py` - 人工上传路由
- `video_resource_routes.py` - 视频资源路由

---

### 2. Services 层

**职责**：
- 实现业务逻辑
- 协调多个核心组件
- 数据转换和验证

**主要服务**：
- `TaskService` - 任务创建、查询、取消
- `FileService` - 文件下载、清理
- `CallbackService` - Webhook 回调
- `NotifyService` - 企业微信通知
- `ManualUploadService` - 人工上传处理
- `MetadataService` - 视频元数据获取

---

### 3. Core 层

**职责**：
- 核心业务逻辑
- 下载器管理
- 异步任务处理

**主要组件**：
- `DownloaderManager` - 下载器管理和降级
- `Worker` - 异步任务队列
- `IPBanBreaker` - IP 熔断器保护

---

### 4. Downloaders 层

**职责**：
- 实现不同下载器
- 统一的下载接口
- 下载器降级和熔断

**下载器**：
- `CDPDownloader` - Chrome DevTools Protocol 下载器
- `YtdlpDownloader` - yt-dlp 本地下载器
- `TikHubDownloader` - TikHub API 服务

---

### 5. Database 层

**职责**：
- 数据持久化
- 数据查询和更新
- 缓存管理

**数据表**：
- `video_resources` - 视频资源（元数据缓存）
- `files` - 文件记录
- `tasks` - 任务记录
- `manual_uploads` - 人工上传记录

## 数据流程

### 下载任务流程

```
1. 客户端创建任务
   POST /api/v1/tasks
   ↓
2. TaskService 创建任务记录
   ↓
3. 检查资源缓存（video_resources）
   ├─ 命中 → 直接返回（cache_hit=true）
   └─ 未命中 → 继续
   ↓
4. Worker 将任务加入队列
   ↓
5. Worker 执行任务
   ↓
6. DownloaderManager 选择下载器
   ├─ CDP（如果启用）
   ├─ yt-dlp
   └─ TikHub
   ↓
7. 下载器下载资源
   ├─ 音频文件 → /data/files/audio/
   └─ 字幕文件 → /data/files/transcript/
   ↓
8. FileService 保存文件记录
   ↓
9. VideoResources 更新缓存
   ↓
10. CallbackService 发送 Webhook（如配置）
   ↓
11. NotifyService 发送企微通知（如配置）
```

## 设计原则

### 1. 低耦合、高内聚

- 各模块职责单一
- 模块间通过接口通信
- 易于测试和维护

### 2. 可扩展性

- 支持添加新下载器
- 支持多种通知方式
- 支持多种存储后端

### 3. 容错性

- 下载器降级机制
- 熔断器保护
- 任务重试机制

### 4. 缓存优化

- 元数据永久缓存
- 文件级缓存复用
- API 响应缓存（TikHub）

### 5. 风控应对

- IP 熔断器保护
- CDP 真实浏览器指纹
- Cookie 管理和协同
- PO Token 支持
